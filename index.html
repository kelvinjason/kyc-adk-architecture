Google ADK 开户审查 (KYC) 系统

完整架构思路与设计哲学

流水线 + 并行 + 护栏

📌 整体设计哲学

整个系统采用 "流水线 + 并行 + 护栏" 三层思维设计：

  • 流水线：保证阶段顺序，前序未完成不进入后序。
  • 并行：保证审查效率，多维度同时进行互不阻塞。
  • 护栏：保证合规安全，横切关注点不侵入业务逻辑。

* ADK 的价值在于这三件事都有原生支持，不需要自己造轮子。

一、主流水线 (SequentialAgent)

三个阶段串行，前序未完成不进入后序：

Stage 1: 入件登记
↓
Stage 1.5: 补件循环（按需）
↓
Stage 2: 五路并行审查
↓
Stage 3: 综合决策

串行的原因：决策依赖审查结果，审查依赖入件数据，顺序不能乱。

二、五路并行审查 (ParallelAgent)

五个维度同时执行，互不阻塞，效率提升约 80%：

审查维度核心关注点
🪪 身份核验证件真伪、活体检测、人脸比对
🛡️ 合规筛查制裁名单、PEP 政治敏感人物、AML 风险评分
💳 信用财务信用报告、财富来源核实
🌍 地理位置FATF 风险国籍/IP 来源矛盾、居住地址一致性
📰 负面新闻Google 搜索欺诈舆情、裁判文书/失信被执行人

三、三个特殊节点

📍 节点 A：补件循环 (LoopAgent)
入件发现缺件 → 通知补件 → 校验质量 → 达标退出 (最多3次)。解决传统流程依赖人工跟进的问题。
📍 节点 B：证件文件管理 (Artifacts)
区分 Session State (存文本JSON结论) 和 Artifacts (专门存二进制文件)。
分为 Session级 (临时图片) 和 User级 (永久合规报告)。
📍 节点 C：人工复核门控 (Human-in-the-loop)
风险 HIGH/CRITICAL 自动暂停 → 推送工单审批 → 录入结果 → 恢复执行。低风险直接穿透。

四、合规护栏层 (Callbacks)

贯穿所有 Agent 的横切关注点，不侵入业务逻辑：

before_agent → 黑名单快速拦截 (命中直接 REJECT)

before_model → PII 脱敏 (身份证号/手机号替换掩码)

before_tool  → 人工审批门控 (高风险暂停)

after_agent  → 写审计日志 (记录谁做了决定，时间戳)

核心价值：监管要求的审计追溯、数据安全、合规拦截，全在护栏层统一处理，业务 Agent 保持干净。

五、数据流设计 (State 分层)

四个作用域各司其职，避免数据泄漏和重复加载：


app 级（全应用共享）
  └── FATF 黑名单列表、全局风险国家库 (定时更新)

user 级（跨会话持久）
  └── 客户历史风险画像、上次审查记录 (开户直接读取)

session 级（本次审查）
  └── 入件数据、五路审查结果、最终决策 (审查完成可归档)

temp 级（单轮临时）
  └── Agent 内部推理中间变量 (不跨轮次自动丢弃)

六、决策规则

五维度综合评分触发条件：

命中制裁/黑名单/重大欺诈🔴 REJECT (Callback 层提前拦截)
在失信/裁判文书/刑事记录命中🔴 REJECT
风险分 ≥ 75 或 2+ 维度复核🟡 MANUAL_REVIEW (触发人工门控)
1 个维度需强化尽调 (EDD)🟡 EDD
1 个维度需关注，其余通过🟡 CONDITIONAL_APPROVE
全部五维度通过✅ APPROVE

八、ADK 特性的必要性

ADK 特性如果不用会怎样
SequentialAgent决策 Agent 可能在审查未完成时就运行
ParallelAgent五路审查串行执行，等待时间线性叠加
LoopAgent补件逻辑需要外部系统驱动，流程断裂
Callbacks审计日志、PII 脱敏分散在各个 Agent 里，难维护
Artifacts证件图片塞进 State，会导致 token 超限崩溃
State 分层历史画像需重查，数据在不该共享的 Agent 间泄漏
DatabaseSession系统重启后所有进行中的审查全部丢失
内置 search需自建搜索代理，维护成本高
