Google ADK 开户审查（KYC）系统

完整架构思路 · Architecture Design

"流水线 + 并行 + 护栏" 三层思维设计

用 Reveal.js 演示 · 按 → 或空格翻页

核心理念三层层级各司其职

  • 流水线（SequentialAgent）
    保证阶段顺序，决策依赖审查，审查依赖入件
  • 并行（ParallelAgent）
    五路审查同时执行，效率提升约 80%
  • 护栏（Callbacks）
    合规拦截、审计追溯、数据安全，横切不侵入

ADK 的价值：这三件事都有原生支持，不需要自己造轮子。

一主流水线（SequentialAgent）

Stage 1: 入件登记
          ↓
Stage 1.5: 补件循环（按需激活，最多 3 轮）
          ↓
Stage 2: 五路并行审查
          ↓
Stage 3: 综合决策

三个阶段串行，前序未完成不进入后序 — 顺序不能乱。

二五路并行审查（ParallelAgent）

🪪 身份核验证件真伪、活体检测、人脸比对
🛡️ 合规筛查制裁名单、PEP、AML 风险评分
💳 信用财务信用报告、财富来源核实
🌍 地理位置FATF 风险国籍、IP/地址矛盾
📰 负面新闻欺诈舆情、裁判文书、失信被执行人

并行效率

串行：5×T
并行：max(T)
提升约 80% 效率

节点 A补件循环（LoopAgent）

发现缺件 → 通知客户补件 → 校验材料质量 → 达标则退出循环

解决的问题：传统流程需要人工跟进补件，LoopAgent 让这个过程完全自动化。

最多循环 3 次，仍未达标则终止本次审查。

节点 B证件文件管理（Artifacts）

两类数据存储

  • Session State
    JSON 结构文本
    审查结论、中间结果
  • Artifacts
    二进制文件
    证件图片、PDF 报告

两套作用域

  • Session 级
    证件图片，审查完可清理
  • User 级（user: 前缀）
    最终 PDF 合规报告
    永久保留，合规存档

节点 C人工复核门控（Human-in-the-loop）

风险 HIGH/CRITICAL
    → 推送工单到合规团队
    → 等待人工审批
    → 录入结果
    → 恢复执行

低风险案件直接穿透，不阻塞

四合规护栏层（Callbacks）

before_agent  → 黑名单快速拦截（命中直接 REJECT，不调 LLM）
before_model  → PII 脱敏（身份证号/手机号替换为掩码）
before_tool   → 人工审批门控（高风险暂停）
after_agent   → 写审计日志（谁做了什么决定，时间戳）

监管要求的审计追溯、数据安全、合规拦截，全部在护栏层统一处理，业务 Agent 保持干净。

五数据流设计（Session State 分层）

app 级（全应用共享）
 └── FATF 黑名单列表、全局风险国家库
     → 所有案件共用，定时更新一次即可

user 级（跨会话持久）
 └── 客户历史风险画像、上次审查记录
     → 再次开户时直接读取，不用重查历史

session 级（本次审查）
 └── 入件数据、五路审查结果、最终决策
     → Agent 间传递，审查完成可归档

temp 级（单轮临时）
 └── Agent 内部推理中间变量
     → 不跨轮次，自动丢弃

六决策规则（五维度综合评分）

🔴 REJECT       任意维度命中制裁/黑名单/重大欺诈（Callback 层提前拦截）
🔴 REJECT       在失信/裁判文书/刑事记录命中
🟡 MANUAL_REVIEW    风险分 ≥ 75 或 2+ 维度复核（触发人工门控）
🟡 EDD            1 个维度需强化尽调（Enhanced Due Diligence）
🟡 CONDITIONAL_APPROVE  1 个维度需关注，其余通过
✅ APPROVE      全部五维度通过

八每项 ADK 特性的必要性

SequentialAgent决策 Agent 可能在审查未完成时就运行
ParallelAgent五路审查串行执行，等待时间线性叠加
LoopAgent补件逻辑需要外部系统驱动，流程断裂
Callbacks审计日志、PII 脱敏分散在每个 Agent 里，难维护
Artifacts证件图片塞进 State，会导致 token 超限崩溃
State 分层历史风险画像每次重查，或数据在不该共享的 Agent 间泄漏
DatabaseSessionService系统重启后所有进行中的审查丢失
内置 google_search需自建搜索代理，维护成本高

总结

八项 ADK 特性合在一起，构成一个生产级合规、可审计、可扩展的开户审查系统。

如需开始写代码实现，可从任意一个 Stage 或 Agent 入手。
