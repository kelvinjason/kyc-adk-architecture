Google ADK 开户审查（KYC）系统

完整架构思路 · Architecture Design

整个系统用 "流水线 + 并行 + 护栏" 三层思维设计：流水线保证阶段顺序，并行保证审查效率，护栏保证合规安全。ADK 的价值在于这三件事都有原生支持，不需要自己造轮子。

一 主流水线（SequentialAgent）

三个阶段串行，前序未完成不进入后序：

Stage 1: 入件登记
         ↓
Stage 1.5: 补件循环（按需）
         ↓
Stage 2: 五路并行审查
         ↓
Stage 3: 综合决策

串行的原因：决策依赖审查结果，审查依赖入件数据，顺序不能乱。

二 五路并行审查（ParallelAgent）

五个维度同时执行，互不阻塞：

审查维度核心关注点
🪪 身份核验证件真伪、活体检测、人脸比对
🛡️ 合规筛查制裁名单、PEP 政治敏感人物、AML 风险评分
💳 信用财务信用报告、财富来源核实
🌍 地理位置FATF 风险国籍/IP 来源矛盾、居住地址一致性
📰 负面新闻Google 搜索欺诈舆情、裁判文书/失信被执行人

并行的价值：五路审查从串行 5×T 压缩到并行 max(T)，效率提升约 80%。

三 三个特殊节点

节点 A：补件循环（LoopAgent）

入件阶段发现材料不完整时激活，最多循环 3 次：

发现缺件 → 通知客户补件 → 校验材料质量 → 达标则退出循环

解决的问题：传统流程需要人工跟进补件，LoopAgent 让这个过程完全自动化。

节点 B：证件文件管理（Artifacts）

区分两类数据存储：

  • Session State → 只存文本型审查结论（JSON 结构）
  • Artifacts → 专门存二进制文件（证件图片、PDF 报告）

两套作用域：

  • Session 级 Artifact：证件图片，审查完可清理
  • User 级 Artifact（user: 前缀）：最终 PDF 合规报告，永久保留，合规存档

节点 C：人工复核门控（Human-in-the-loop）

高风险案件在 decision_agent 执行前自动暂停：

风险 HIGH/CRITICAL → 推送工单到合规团队 → 等待人工审批 → 录入结果 → 恢复执行
低风险案件直接穿透，不阻塞。

四 合规护栏层（Callbacks）

贯穿所有 Agent 的横切关注点，不侵入业务逻辑：

每个 Agent 执行前   → 黑名单快速拦截（命中直接 REJECT，不调 LLM）
每次调 LLM 前       → PII 脱敏（身份证号/手机号替换为掩码）
每次调工具前        → 人工审批门控（高风险暂停）
每个 Agent 完成后   → 写审计日志（谁做了什么决定，时间戳）

Callbacks 的核心价值：监管要求的审计追溯、数据安全、合规拦截，全部在护栏层统一处理，业务 Agent 保持干净。

五 数据流设计（Session State 分层）

四个作用域各司其职，避免数据泄漏和重复加载：

app 级（全应用共享）
 └── FATF 黑名单列表、全局风险国家库
     → 所有案件共用，定时更新一次即可

user 级（跨会话持久）
 └── 客户历史风险画像、上次审查记录
     → 再次开户时直接读取，不用重查历史

session 级（本次审查）
 └── 入件数据、五路审查结果、最终决策
     → Agent 间传递，审查完成可归档

temp 级（单轮临时）
 └── Agent 内部推理中间变量
     → 不跨轮次，自动丢弃

六 决策规则（五维度综合评分）

🔴 REJECT    任意维度命中制裁/黑名单/重大欺诈   （Callback 层可提前拦截）
🔴 REJECT    在失信/裁判文书/刑事记录命中
🟡 MANUAL_REVIEW    风险分 ≥ 75 或 2+ 维度复核      （触发人工门控）
🟡 EDD         1 个维度需强化尽调（EDD）
🟡 CONDITIONAL_APPROVE  1 个维度需关注，其余通过
✅ APPROVE    全部五维度通过

八 每项 ADK 特性的必要性

ADK 特性如果不用会怎样
SequentialAgent决策 Agent 可能在审查未完成时就运行
ParallelAgent五路审查串行执行，等待时间线性叠加
LoopAgent补件逻辑需要外部系统驱动，流程断裂
Callbacks审计日志、PII 脱敏分散在每个 Agent 里，难维护
Artifacts证件图片塞进 State，会导致 token 超限崩溃
State 分层历史风险画像每次重查，或数据在不该共享的 Agent 间泄漏
DatabaseSessionService系统重启后所有进行中的审查丢失
内置 google_search需自建搜索代理，维护成本高

本文档为纯架构设计思路，不包含实现代码。
